NAT traversal: el problema que todo desarrollador multiplayer debe entender

Julio 2026 · Redes

Has creado un juego multiplayer. En tu casa, con dos ventanas del navegador abiertas, funciona perfecto. Pero cuando tu amigo intenta conectarse desde su casa, no pasa nada. El culpable más probable es el NAT.

¿Qué es NAT?

NAT (Network Address Translation) es el mecanismo que permite que múltiples dispositivos en tu casa compartan una sola dirección IP pública. Tu router asigna direcciones IP privadas (como 192.168.1.x) a cada dispositivo, y cuando estos se conectan a Internet, el router traduce las direcciones privadas a la IP pública.

El problema: cuando otro dispositivo en Internet intenta iniciar una conexión con tu computadora, el router no sabe a cuál de tus dispositivos internos enviar el paquete. Es como tener un edificio con un portero que solo deja salir, pero no entra nada que no haya sido solicitado.

Los tres protocolos que resuelven el problema

STUN (Session Traversal Utilities for NAT)

STUN es un servidor público que hace una tarea simple: cuando un cliente le envía un paquete, STUN responde diciéndole "esta es tu IP pública y este es tu puerto". El cliente descubre así su dirección reflexiva (la que el NAT expuso para esa sesión).

STUN es barato y eficiente. Un solo servidor puede manejar decenas de miles de clientes. Pero no siempre funciona: cuando ambos clientes están detrás de NATs simétricos (como los de redes móviles), STUN falla.

TURN (Traversal Using Relays around NAT)

Cuando STUN no es suficiente, entra TURN. TURN es un relay: ambos clientes envían su tráfico a un servidor TURN, y este reenvía los paquetes al destino. Es más caro (pagas por cada byte) y añade latencia, pero siempre funciona.

Entre el 8% y el 15% de las conexiones de consumidores necesitan TURN en la práctica.

ICE (Interactive Connectivity Establishment)

ICE es el protocolo que orquesta todo. Cada cliente recolecta candidatos de conexión:

  • Host — la IP local (LAN)
  • Server Reflexive — la IP pública descubierta via STUN
  • Relayed — la dirección del relay TURN

Los candidatos se intercambian a través de un canal de señalización (normalmente un WebSocket). Luego, cada par prueba las combinaciones en orden de prioridad: primero directo (host-host), luego reflexivo-reflexivo, y como último recurso, relay.

Keepalives: el detalle que muchos olvidan

Los NATs tienen memoria: mantienen una tabla de traducción solo mientras hay tráfico activo. Si un cliente queda inactivo por 30 segundos, la tabla expira y los paquetes entrantes se pierden. La solución es enviar un keepalive (un paquete minúsculo) cada 15-20 segundos en cada sesión activa.

Implementación práctica para tu juego

Si estás desarrollando un juego multiplayer en JavaScript, la forma más directa de implementar todo esto es usar WebRTC, que incluye STUN, TURN e ICE de forma nativa:

const config = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    { urls: 'turn:tu-servidor.com:3478',
      username: 'user',
      credential: 'pass' }
  ]
};
const peerConnection = new RTCPeerConnection(config);

Resumen

  • NAT permite compartir IPs pero bloquea conexiones entrantes
  • STUN descubre tu IP pública
  • TURN actúa como relay cuando STUN no basta
  • ICE orquesta la mejor ruta posible
  • Siempre implementa keepalives para mantener la sesión activa
